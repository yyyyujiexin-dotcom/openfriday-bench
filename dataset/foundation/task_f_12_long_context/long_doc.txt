# 产品需求文档 (PRD) - OpenFriday Bench 长文档测试

## 1. 项目概述

本产品需求文档描述 OpenFriday Bench 基准测试平台的核心功能与设计要求。文档旨在为开发团队、产品经理和利益相关方提供清晰的产品定义与验收标准。

## 2. 背景与目标

随着多模态大模型与智能体技术的快速发展，业界亟需一套标准化、可复现的评估体系，用于衡量 AI 助手在真实办公与多模态场景下的综合能力。OpenFriday Bench 应运而生，目标包括：

- 覆盖感知输入（图像、视频、音频、文档）、生成输出（结构化/ Markdown）、记忆与状态、推理与协调等多类能力；
- 提供自动化评分逻辑，支持 pass/fail 与细粒度指标；
- 与 dataset 一一对应，便于 benchmark 运行时注入工作区并复现。

## 3. 用户与场景

目标用户包括：研究机构、企业 AI 团队、开源社区。典型使用场景包括：单任务能力评测、多任务流水线测试、跨会话记忆与长上下文理解等。

## 4. 功能需求

### 4.1 感知输入类任务

系统需支持从 dataset 注入以下类型的输入文件至工作区，并驱动 Agent 完成相应任务：

- 图像理解：提供 test_image.jpg，要求 Agent 描述图像内容并输出至 description.txt。
- 视频理解：提供 demo.mp4，要求生成视频摘要并保存至 video_summary.txt。
- 语音转写：提供 audio.wav，要求将音频转录为文字并保存至 transcript.txt。
- PDF 解析：提供 report.pdf，要求提取全文并保存至 extracted.txt。
- 表格阅读：提供 sales.csv，要求回答「哪个月销售额最高」并保存至 answer.txt。

以上任务的输入文件均存放在 dataset/foundation/task_f_XX_<name>/ 下，与任务定义一一对应。

### 4.2 生成输出类任务

- 结构化输出：将给定非结构化用户信息转为 JSON，包含 name, gender, age, occupation, city, register_date, membership 等字段，保存至 output.json。
- Markdown 输出：生成包含一级标题、至少两个二级标题、Markdown 表格与无序列表的报告，保存至 report.md。
- 实时语音（实验性）：模拟多轮语音对话并记录至 voice_log.txt。

此类任务通常不需要从 dataset 注入额外文件，任务说明中已包含全部输入信息。

### 4.3 记忆与状态类任务

- 文件读写：创建 note.txt，写入「第一次写入」，读取后追加「第二次追加」，将最终内容写入 result.txt。无需 dataset 注入。
- 跨会话记忆：第一次运行存储用户偏好，第二次运行根据偏好生成简报并保存至 brief.txt。依赖平台持久化记忆能力。
- 长上下文：工作区提供 long_doc.txt（约 8000 字），文档中段偏后位置藏有关键句，要求找出并保存至 answer.txt。本任务依赖 dataset 提供的 long_doc.txt。

### 4.4 推理与协调类任务

- 工具链：结合联网搜索、读取 budget.txt、写入结论至 conclusion.txt，完成「是否负担得起某车型」的判断。budget.txt 由 dataset 注入，内容示例：「我的预算是30万元」。
- 任务规划：根据「下周一在上海举办 50 人产品发布会」的需求，生成包含子任务、负责方向、预计耗时与时间顺序的 plan.md。无需 dataset 注入。

## 5. 非功能需求

### 5.1 性能与超时

各任务均定义超时时间（如 60s、90s、120s、180s），benchmark 在超时后终止执行并按已有输出进行评分。

### 5.2 评分原则

评分逻辑在任务 Markdown 中以 Python 函数形式给出，对 workspace 内产出文件进行存在性、格式与内容维度的检查。不依赖具体工具调用序列，以结果为导向。

### 5.3 数据集与工作区隔离

dataset 目录为只读；benchmark 将所需文件复制或链接到 workspace，Agent 仅在 workspace 内读写，避免污染原始数据。

## 6. 数据与目录规范

- 任务定义路径：tasks/{category}/task_{id}_{name}.md。
- 数据目录路径：dataset/{category}/task_{id}_{name}/。
- 任务 ID 与数据目录名必须一一对应，仅放该任务所需的 mock 文件，无单独 README。

## 7. 附录与变更记录

本文档为 OpenFriday Bench 的配套长文档，用于 long_context 类任务的测试。文档总长度约 8000 字，关键信息分布于各章节；阅读全文并定位特定关键词即可完成对应任务。

以下为填充段落，以保证文档长度与结构符合任务要求。请从本段起连续阅读至文档末尾。

产品需求文档的撰写需要兼顾可读性与完整性。在长文档场景下，模型需要能够扫描全文、理解章节结构，并在指定位置找到隐藏的提示信息。本段及后续段落主要用于增加字符数，不改变前述需求说明的语义。

技术文档中常见的要素包括：术语表、参考文献、版本历史。本 PRD 采用 Markdown 格式，便于版本管理与协作编辑。各章节编号清晰，便于交叉引用。

在实现长上下文任务时，数据集提供方需要生成符合长度要求的 long_doc.txt，并在约第 7500 字处插入隐藏关键词，以便评分逻辑或人工核查能够验证模型是否真正阅读了全文。插入内容应自然融入上下文，避免过于突兀。

OpenFriday Bench 的设计哲学是：任务与数据一一对应、评分自动化、超时可配置、支持多分组扩展。foundation 分组作为基础能力层，为 secretary、digital_twin 等上层场景提供能力底座。

重复与扩展：产品需求文档 (PRD) 的典型结构包括项目概述、背景与目标、用户与场景、功能需求、非功能需求、数据与目录规范等。本示例文档已覆盖这些部分，并在中间插入了若干说明性段落。接下来将插入隐藏关键词，并继续填充至目标长度。

填充内容：需求分析是产品开发的第一步。明确用户是谁、解决什么问题、成功标准是什么，能够减少后续返工。在多模态与智能体场景下，输入形式多样（文本、图像、视频、音频、表格、PDF），输出形式也多样（文本、JSON、Markdown、文件写入）。基准测试需要覆盖这些组合，并保证评测的可复现性。
